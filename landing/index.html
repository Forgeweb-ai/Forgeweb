// How it works

The whole picture, in one diagram.

Forge is four moving parts: a browser-side UI, an orchestrator, a sandboxed agent container, and a logging proxy in front of your LLM provider. Everything else (Postgres, the per-project workspace, the snapshot store) hangs off these.

A. What happens when you hit send

Your message goes to the AI agent. The agent reads and writes files in your project folder, runs commands, and asks your chosen model what to do next. Each request to the model is logged on your machine so you can see exactly what the agent sent — handy when you're paying per token.

B. Every project has a real folder

Each project lives on disk as a normal folder of files — open it in your editor, run commands in it, commit it to git if you want. Forge keeps a snapshot every time the AI takes a turn, so if an edit goes wrong you can roll back without losing anything you typed.

C. Your account, your keys, your data

Your account, your project list, and every provider key you've added live in a Supabase that runs on your machine. The installer brings it up for you — there is no hosted database, no Forge-controlled server holding your data. Provider keys are encrypted before they hit disk; decryption only happens on the exact provider call that needs them.

Separately, the apps you build can connect their own Supabase via in-app OAuth if they need auth, a database, or file storage of their own. That's distinct from the Supabase that runs Forge itself.

D. Self-host only (today)

Forge is open source and runs entirely on your machine. There is no Forge SaaS — no "hosted" tier, no account on a Forge server, no telemetry phoned home. The same Docker stack you spin up locally is the whole product.

For the apps your AI builds, Forge provisions a local Postgres schema by default. If those apps need real auth, file storage, or a public database, connect a Supabase project via the in-app OAuth flow — Forge stores a scoped delegation token, never your service_role JWT.

E. See exactly what the AI is sent

Every request to your AI provider is written to a local log folder before it goes out. If you ever wonder "what did the agent send the model with my key?" — open the folder, the answer is right there in plain JSON. You don't have to take anyone's word for it.

F. AI images that don't get lost

When the agent wants to drop an AI-generated image into your app, it queues the job and shows a placeholder until the image arrives. If your image provider is busy, Forge retries in the background instead of giving up — so a temporary rate limit doesn't leave you with broken pictures in the page.

// BYOK

What you bring (or don't).

Forge starts free — the default model is opencode zen's DeepSeek V4 Flash Free, no key required. If you want a paid model, you connect your own provider, pay your own bill, and cancel any time by removing the key from Settings. Forge doesn't resell tokens or hold a platform key on your behalf.

Note: the database that holds your account and your keys (Supabase, on your machine) is set up by the installer — you don't bring it, but it's not optional either. It's part of what makes Forge run.

// Install · self-host

Two ways to run it. Both stay on your machine.

Forge is open source and self-hosted — there is no SaaS yet. Everything you do runs on your own laptop or server, including the AI agent. The default model is DeepSeek V4 Flash Free via opencode zen, so a fresh install works with zero keys. Bring an Anthropic / OpenAI / Moonshot / Google / Replicate key from in-app Settings if you want a paid model. Keys never live in .env — they're stored encrypted on your machine and only sent on the exact provider request that needs them. Prereqs: Docker (Desktop or Engine + Compose v2), ~4GB RAM, ~10GB disk. macOS, Linux, WSL2.

Both paths bring up the whole stack — the chat UI, the agent, the audit-log proxy, and a local Supabase that holds your account, project list, and encrypted provider keys. None of that needs to be configured; it just comes up.

Path A — Docker image (recommended)

For most users. Pulls prebuilt images from the registry — no source clone, no Python, no bun, no build step. Two commands.

  1. Fetch the compose file

    curl -fsSL https://raw.githubusercontent.com/Forgeweb-ai/Forgeweb/main/docker-compose.release.yml \
  -o docker-compose.yml

    The release compose file uses published ghcr.io/forgeweb/* images instead of building from source.

  2. Bring up the stack

    docker compose up -d

    First boot pulls ~1.2 GB of images (Postgres, Redis, Traefik, forge-server, forge-ui, opencode, forge-llm-proxy) and brings everything up behind Traefik on *.forge.localhost. Allow ~2 minutes; subsequent boots are near-instant from the local image cache.

  3. Open the app and sign up

    Navigate to http://app.forge.localhost and create an account. The wildcard *.localhost resolves to 127.0.0.1 on every major OS — no /etc/hosts edit.

  4. Start building — no key required

    Type a prompt. The default DeepSeek V4 Flash Free model responds with zero setup.

    Want a paid model (Claude Sonnet, GPT-Image, Flux, etc.)? Open Settings → API Keys and paste a provider key. Switch models from the composer dropdown — no restart.

  5. (Optional) Connect Supabase for the apps you build

    Forge already runs on its own local Supabase (your account, project list, and encrypted keys live there). This step is different: it lets the apps you build get their own Supabase project for auth / database / storage. From Settings, click Connect Supabase and complete the OAuth handoff — Forge holds a scoped delegation token, never your service_role JWT.

Path B — From source (for contributors)

For developers who want to modify Forge itself. Clones the repo, builds from source, and starts everything via the dev scripts.

  1. Clone the repo

    git clone https://github.com/Forgeweb-ai/Forgeweb.git
cd forge

  2. Run the installer

    One script. It detects your OS, verifies Docker is running, checks Python 3.11+ and bun (auto-installs bun if missing), confirms ~10 GB free disk, generates a JWT secret, and hands off to dev.sh to bring the stack up.

    ./install.sh

    Idempotent — re-run any time. ./install.sh --check runs a pre-flight report without changing anything. Daily restart: ./dev.sh.

  3. Open the app

    Same URLs as Path A: http://app.forge.localhost. Sign up, start building, optionally add a paid-model key in Settings.

// Documentation

The whole stack, explained.

Click any section to expand. We keep these inline so you don't have to leave the page to evaluate Forge. If you want to read the whole thing top-to-bottom, expand them all — the order matches how the system loads on first boot.

1. forge-ui — the chat front end

This is the app you actually use. It shows the chat composer, the running preview of what you're building, the file tree of the project, and Settings. It only talks to forge-server — it never calls an LLM directly, so a leaky front-end can't expose your provider key.

The visual language is the same warm-cream palette in light mode and a deep-navy palette in dark — you're looking at it right now. Both modes share the same tokens, so anything you customize looks consistent across the app.

2. forge-server — the orchestrator

The brain. It owns your account, your projects, your settings, and the key vault. When you type a message, forge-server is what decides which AI model to call, hands the agent the right per-user key, kicks off image jobs, and starts a fresh preview container for your project if one isn't already running.

Idle previews go to sleep after a few minutes of inactivity so you don't have a dozen project containers eating memory. They wake the moment you hit the URL again.

3. opencode — the agent that actually edits your code

We vendor the opencode repo into opencode/. It runs as its own container (one container, shared by all projects; isolation between projects is at the workspace-path level) and exposes a small HTTP API that forge-server drives.

On top of opencode, Forge adds the things a webapp builder needs: per-user keys, Supabase wiring, AI image generation, a design system for the AI to pick from, and a verify step that runs after every turn to catch breakage before you see it. Everything we add to opencode itself stays under opencode's MIT license — see the NOTICE file.

4. The audit log for every AI call

Every request your agent makes to a paid model is recorded to a local folder before it leaves your machine. Open forge-llm-proxy-logs/, find the call, read what was sent and what came back. Nothing is hidden — and if a provider stops streaming mid-response, the log shows that too instead of hanging the chat indefinitely.

5. Each project gets its own live preview

Every project you create gets a folder on disk and a running preview URL of its own (something like my-app.preview.forge.localhost). You can browse it like a real site while you're building. Projects you haven't touched in a while go to sleep automatically and wake the moment you visit them again — so you can have dozens of projects without your laptop fans spinning.

6. Versioning — a snapshot every AI turn

Forge saves a version of every file in your project each time the AI takes a turn. If a turn breaks something, you can roll back to any earlier point with one click — no merging, no undo-history hunting. You don't have to commit anything to git for this to work; the snapshots are separate from your git history (if you use git at all).

7. AI image generation

When the agent decides your page needs an image, it asks your image provider (Flux, gpt-image-1, Imagen, etc.) in the background and drops a placeholder in the page until the real image arrives. If your provider is busy, Forge retries quietly instead of leaving broken images sitting in your page.

If you haven't added an image-model key, the agent falls back to a stock-photo search so your page still has art — just not custom-generated art.

8. Self-host only — and the two Supabases

Forge is open source and runs on your machine. There is no hosted tier, no Forge-controlled account server, and no telemetry. The whole product is what you start with docker compose up. If you want apps you build to be reachable from the public internet, you deploy them yourself (Vercel, Fly, your own server) — Forge doesn't host them for you.

There are two Supabases in the story — worth keeping straight:

1. Forge's Supabase — runs on your machine, set up automatically by the installer. This is where your Forge account lives, where your project list lives, and where your encrypted provider keys live. It's not optional; it's how Forge works. You never connect to a hosted Forge database because there isn't one.

2. The Supabase your apps connect to (optional) — for the apps you build that need their own auth, database, or file storage. You connect this from in-app Settings via OAuth, and Forge holds a scoped delegation token (never your service_role JWT) to provision and wire it into the apps that need it. This Supabase is yours, lives wherever you keep it, and is fully separate from Forge's own storage. The two never mix.
